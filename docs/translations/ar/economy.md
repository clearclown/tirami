# Forge — النموذج الاقتصادي (Economic Model)

## معيار الحوسبة (Compute Standard - 計算本位制)

كل نظام نقدي مدعوم بالندرة (scarcity):

| العصر | المعيار | الدعم |
|-----|----------|---------|
| القديم | الذهب/الفضة | ندرة جيولوجية |
| 1944–1971 | بريتون وودز | الدولار مرتبط بالذهب |
| 1971–الحاضر | البترودولار | الطلب على النفط + القوة العسكرية |
| 2009–الحاضر | البيتكوين | الكهرباء المحروقة في SHA-256 |
| **Forge** | **معيار الحوسبة** | **الكهرباء المنفقة على استنتاج مفيد** |

يقدم Forge معيار الحوسبة: وحدة القيمة مدعومة بإنفاق طاقة حقيقي لأداء حوسبة مفيدة. على عكس إثبات العمل في بيتكوين، كل جول يُنفق في Forge ينتج ذكاءً حقيقياً.

## وحدات الحوسبة (CU): العملة الأصلية

### ما هي وحدات الحوسبة (CU)

**1 CU = مليار عملية ملموسة (FLOPs) من عمل الاستنتاج المحقق.**

CU ليست عملة مشفرة (cryptocurrency). وليست توكناً على بلوكشين. إنها وحدة حساب تمثل حوسبة حقيقية تم إجراؤها. لـ CU قيمة لأنها مطالبة بحوسبة مستقبلية — إذا كسبت CU عن طريق تقديم الاستنتاج، يمكنك إنفاقها لتلقي الاستنتاج.

### لماذا CU وليس البيتكوين

| الخاصية | CU | البيتكوين |
|----------|-----|---------|
| **دعم القيمة** | حوسبة مفيدة (جوهرية) | حوسبة هاش (مصطنعة) |
| **سرعة التسوية** | فورية (دفتر محلي) | ثوانٍ إلى دقائق (Lightning/سلسلة) |
| **تكلفة المعاملة** | صفر | رسوم القنوات، رسوم السلسلة |
| **التبعية الخارجية** | لا يوجد | صحة شبكة البيتكوين |
| **خطر الكوانتم** | لا يوجد (لا لغز تشفيري) | SHA-256 / ECDSA معرضة للخطر |
| **توليد العائد** | نعم (العتاد الخامل يكسب CU) | لا (البيتكوين في المحفظة لا يكسب شيئاً) |

CU هي وحدة التسوية **الأساسية**. البيتكوين، العملات المستقرة، والعملات النقدية هي **مخارج (off-ramps)** اختيارية متاحة عبر محولات الجسور خارج البروتوكول.

### CU كأصل إنتاجي

```
مبنى سكني                   Mac Mini على Forge
───────────────────         ──────────────────
الأصل: المبنى               الأصل: عتاد الحوسبة
التكلفة: الصيانة            التكلفة: الكهرباء
الإيراد: الإيجار            الإيراد: CU من الاستنتاج
العائد: الإيجار - الصيانة   العائد: CU المكتسبة - تكلفة الكهرباء
الخمول = دخل ضائع           الخمول = إمكانات مهدورة
```

جهاز الحوسبة على Forge ليس مثل البيتكوين في المحفظة (قيمة ثابتة، لا عائد). إنه يشبه العقار المستأجر — يولد دخلاً من خلال العمل المفيد.

## نموذج المعاملات

### تنفيذ الصفقات

كل استنتاج ينشئ صفقة بين طرفين:

```rust
pub struct TradeRecord {
    pub provider: NodeId,       // من قام بتشغيل الاستنتاج
    pub consumer: NodeId,       // من طلبه
    pub cu_amount: u64,         // كمية CU المنقولة
    pub tokens_processed: u64,  // العمل المنجز
    pub timestamp: u64,
    pub model_id: String,
}
```

يتم تسجيل الصفقة من قبل كلا الطرفين. في التنفيذ الحالي، تحتفظ كل عقدة بدفتر حسابات محلي. التنفيذ المستهدف يضيف التوقيعات المزدوجة ومزامنة Gossip.

### التسعير الديناميكي

تتغير أسعار CU بناءً على العرض والطلب المحلي:

```
effective_price = base_cu_per_token × demand_factor / supply_factor
```

- **عقد خاملة أكثر** ← يرتفع عامل العرض ← ينخفض السعر.
- **طلبات استنتاج أكثر** ← يرتفع عامل الطلب ← يرتفع السعر.
- كل عقدة تلاحظ ظروف السوق الخاصة بها. لا يوجد دفتر أوامر عالمي.

### الفئة المجانية (Free Tier)

العقد الجديدة التي ليس لها تاريخ مساهمة تحصل على 1,000 CU. هذا يسمح لأي شخص باستخدام الشبكة فوراً. تُستهلك الفئة المجانية من الطلب الأول — ولا يتم تجديدها.

تخفيف هجمات Sybil: إذا ظهرت أكثر من 100 عقدة غير معروفة دون مساهمة، يتم رفض طلبات الفئة المجانية الجديدة.

## إثبات العمل المفيد (Proof of Useful Work)

### المفهوم

إثبات العمل في بيتكوين: "لقد حرقت الكهرباء لحساب هاشات SHA-256. ها هو الرقم العشوائي (nonce) الذي يثبت ذلك."

إثبات العمل المفيد في Forge: "لقد حرقت الكهرباء لتشغيل استنتاج LLM. ها هي الاستجابة، وها هو توقيع المستهلك الذي يؤكد استلامها."

الفرق الجوهري: إثبات البيتكوين ذاتي التوليد (يمكن لأي معدن إنتاج هاش صالح). إثبات Forge يتطلب **طرفاً مقابلاً** — شخصاً أراد الاستنتاج فعلياً. لا يمكنك تزييف الطلب.

### بروتوكول التحقق (المستهدف)

```
1. المستهلك يرسل طلب استنتاج للمزود
2. المزود ينفذ الاستنتاج، يتدفق التوكنات للمستهلك
3. المستهلك يستلم التوكنات، يحسب هاش الاستجابة
4. كلا الطرفين يوقعان سجل الصفقة (TradeRecord):
   - المزود يوقع: "لقد حسبت هذا"
   - المستهلك يوقع: "لقد استلمت هذا"
5. سجل الصفقة الموقع مزدوجاً ينتشر عبر الشبكة (Gossip sync)
6. يمكن لأي عقدة التحقق من كلا التوقيعين
```

لا يمكن للعقدة تضخيم رصيد CU الخاص بها دون طرف مقابل متعاون. التواطؤ ممكن لكنه غير عقلاني اقتصادياً — المستهلك المتواطئ لا يكسب شيئاً من توقيع صفقات وهمية.

### التنفيذ الحالي

يستخدم التنفيذ المرجعي الحالي دفاتر حسابات محلية مع حماية التكامل HMAC-SHA256. التوقيعات المزدوجة و Gossip هي الخطوة التالية.

## العائد والسمعة (Yield and Reputation)

### العائد (Yield)

العقد التي تظل متصلة وتساهم بالحوسبة تكسب عائداً:

```
yield_cu = contributed_cu × 0.001 × reputation × uptime_hours
```

عند مستوى سمعة 1.0، العقدة التي ساهمت بـ 10,000 CU تكسب 80 CU لكل ليلة (8 ساعات). هذا ليس تضخماً — بل مكافأة على التوافر. العقد التي تتصل بشكل موثوق هي أكثر قيمة للشبكة.

### السمعة (Reputation)

كل عقدة لها درجة سمعة بين 0.0 و 1.0:

- العقد الجديدة تبدأ بـ 0.5.
- وقت الاتصال (Uptime) والصفقات الناجحة تزيد السمعة.
- الانفصالات والصفقات الفاشلة تقلل السمعة.
- سمعة أعلى ← معدل عائد أعلى، أولوية في الجدولة.

## التسوية والجسور الخارجية

### القاعدة الأساسية

**التسوية في البروتوكول تكون بوحدات CU.** التحويل إلى أي شيء آخر هو شأن يتعلق بالتكامل.

### بيانات التسوية

يمكن للمشغلين تصدير تاريخ صفقات قابل للتدقيق لأي نافذة زمنية:

```
forge settle --hours 24 --price 0.05 --out settlement.json
```

يتضمن البيان: إجمالي CU المكتسبة، إجمالي CU المنفقة، صافي CU، عدد الصفقات، وسعر مرجعي اختياري لكل CU.

### معمارية الجسور (Bridge Architecture)

```
الطبقة 0: بروتوكول Forge
  → محاسبة CU، الصفقات، التسعير

الطبقة 1: بيان التسوية
  → تاريخ صفقات قابل للتصدير
  → سعر صرف مرجعي

الطبقة 2: الجسر الخارجي (اختياري)
  → CU ↔ BTC (Lightning)
  → CU ↔ عملة مستقرة
  → CU ↔ نقد
```

طبقة الجسر تقع خارج البروتوكول. يمكن للمشغلين المختلفين استخدام جسور مختلفة. يظل البروتوكول مفيداً حتى مع انعدام السيولة الخارجية.

### جسر لابتكوين (Lightning Bridge)

للمشغلين الذين يريدون تسوية بالبيتكوين:

```bash
forge settle --hours 24 --pay
```

هذا ينشئ فاتورة Lightning BOLT11 لصافي CU المكتسبة، محولة بسعر الصرف المكون (الافتراضي: 10 msats لكل CU).

## ميزانيات موجهة من الوكلاء (Agent-Directed Budgets)

### الرؤية

التقليدي: الإنسان يقرر ← الإنسان يدفع ← الذكاء الاصطناعي ينفذ
Forge: السياسة تسمح للوكيل ← الوكيل يتحقق من الميزانية ← الوكيل ينفق CU ← الوكيل ينفذ

### واجهة البرمجة (API)

```
GET /v1/forge/balance   → رصيد CU، المساهمة، الاستهلاك، السمعة
GET /v1/forge/pricing   → سعر السوق، تقديرات التكلفة لكل 100/1000 توكن
```

يمكن للوكيل:
1. التحقق من رصيده قبل إرسال الطلب.
2. تقدير تكلفة الاستنتاج بأسعار السوق الحالية.
3. تقرير ما إذا كان الطلب يستحق تكلفة CU.
4. التنفيذ والدفع تلقائياً.

يحدد المشرفون البشريون سياسات الميزانية. ويعمل الوكلاء ضمن تلك الحدود بشكل مستقل.

### حلقة التعزيز الذاتي

```
وكيل (صغير، هاتف)
  ← يكسب CU عن طريق إقراض الحوسبة الخاملة
  ← ينفق CU للوصول إلى نماذج أكبر
  ← يصبح أذكى
  ← يتخذ قرارات اقتصادية أفضل
  ← يكسب المزيد من CU
  ← يصل إلى نماذج أكبر بكثير
  ← ...
```

هذا نمط تطبيق ممكن. يوفر البروتوكول السوق؛ ويوفر الوكلاء الاستراتيجية.

## لماذا هذا ليس ويب 3 (Web3)

تخلق معظم مشاريع ويب 3 ندرة مصطنعة (توكنات) فوق سلع رقمية وفيرة. يقوم Forge بالعكس:

- **الحوسبة نادرة فعلياً** — تتطلب كهرباء حقيقية، سيليكون حقيقي، ووقت حقيقي.
- **CU ليست للمضاربة** — تمثل عملاً محققاً، وليست رهاناً على التبني المستقبلي.
- **لا طرح أولي (ICO)، لا بيع توكنات، لا توكن حوكمة** — يتم كسب CU بالعمل.
- **لا حاجة لبلوكشين** — التوقيعات الثنائية و Gossip كافية.
- **لا عقود ذكية** — البروتوكول هو العقد.

القيمة تُصنع بواسطة الفيزياء، وليس بالإجماع.
