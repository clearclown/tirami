# Forge — خارطة الطريق (Roadmap)

## المرحلة 1: الاستنتاج المحلي ✅

- `forge-core`: نظام الأنواع (NodeId, LayerRange, ModelManifest, PeerCapability)
- `forge-infer`: محرك llama.cpp، محمل GGUF، توليد التوكنات بالتدفق (streaming)
- `forge-node`: واجهة برمجة تطبيقات HTTP (/chat, /chat/stream, /health)
- `forge-cli`: أمر `forge chat` مع تنزيل النماذج تلقائياً

## المرحلة 2: بروتوكول P2P ✅

- `forge-net`: نقل Iroh، تشفير Noise، اتصالات الأقران
- `forge-proto`: 14 نوعاً من رسائل البروتوكول (bincode + بادئة الطول)
- `forge-node`: خط أنابيب Seed/Worker، طلب/استجابة الاستنتاج
- اختبارات التكامل: عقدتان تتبادلان رسالة Hello + رسائل متعددة

## المرحلة 3: الاستنتاج عن بعد + دفتر حسابات المشغل ✅

- `forge-ledger`: محاسبة CU، تنفيذ الصفقات، السمعة، العائد، تسعير السوق
- `forge-node`: دفتر الحسابات مدمج في خط أنابيب الاستنتاج
- فحوصات رصيد CU قبل الاستنتاج
- سجلات الصفقات بعد الاكتمال
- حماية تكامل دفتر الحسابات باستخدام HMAC-SHA256

## المرحلة 4: واجهة البرمجة الاقتصادية ✅

- واجهة برمجة متوافقة مع OpenAI: `POST /v1/chat/completions`, `GET /v1/models`
- قياس CU: كل استنتاج يسجل صفقة مع ملحق `x_forge`
- نقاط نهاية ميزانية الوكلاء: `GET /v1/forge/balance`, `GET /v1/forge/pricing`
- جسر تسوية CU←Lightning: `forge settle --pay`
- حل النماذج تلقائياً من HF Hub في وضع Seed
- إغلاق آمن عبر Ctrl-C مع حفظ دفتر الحسابات

## المرحلة 5: تكامل فرع mesh-llm (التالي)

**الهدف:** استبدال طبقة الاستنتاج في Forge بمحرك mesh-llm الموزع والمثبت.

| المخرج | الوصف |
|---|---|
| فرع mesh-llm | إنشاء forge كفرع من mesh-llm مع طبقة اقتصادية |
| دمج forge-ledger | ربط تسجيل CU بخط أنابيب الاستنتاج في mesh-llm |
| الحفاظ على API الاقتصادي | الإبقاء على نقاط نهاية /v1/forge/* في الكود الجديد |
| امتداد وحدة تحكم الويب | إضافة رصيد CU ورؤية الصفقات إلى وحدة تحكم mesh-llm |
| Pipeline + MoE | وراثة توازي خط الأنابيب وتجزئة الخبراء من mesh-llm |
| اكتشاف Nostr | وراثة اكتشاف الشبكة العامة من mesh-llm |
| CREDITS.md | توثيق الفضل لـ mesh-llm |

## المرحلة 6: إثبات العمل المفيد (Proof of Useful Work)

**الهدف:** جعل مطالبات CU قابلة للتحقق عبر الشبكة.

| المخرج | الوصف |
|---|---|
| بروتوكول التوقيع المزدوج | كلا المزود والمستهلك يوقعان كل سجل صفقة |
| مزامنة Gossip | الصفقات الموقعة تنتشر عبر الشبكة |
| كشف الاحتيال | رفض الصفقات غير الموقعة أو غير المتطابقة |
| نشر السمعة | مشاركة درجات السمعة عبر الأقران |
| مقاومة التواطؤ | كشف النماذج الشاذة إحصائياً في أنماط الصفقات |

## المرحلة 7: الجسور الخارجية (External Bridges)

**الهدف:** السماح للمشغلين بتحويل CU إلى قيمة خارجية.

| المخرج | الوصف |
|---|---|
| جسر Lightning | تسوية تلقائية CU→sats عبر LDK |
| محول العملات المستقرة | تحويل CU→USDC/USDT |
| واجهة محول النقد | مواصفات لتسوية التحويلات البنكية |
| خدمة سعر الصرف | تغذيات عامة لأسعار CU/BTC و CU/USD |
| التثبيت في بيتكوين | اختياري: جذر Merkle دوري ← OP_RETURN لمسار تدقيق غير قابل للتغيير |

## المرحلة 8: اقتصاد الوكلاء المستقلين (Agent Autonomous Economy)

**الهدف:** السماح لوكلاء الذكاء الاصطناعي بإدارة دورة حياة الحوسبة الخاصة بهم.

| المخرج | الوصف |
|---|---|
| سياسات الميزانية | حدود إنفاق يحددها البشر لكل وكيل |
| التداول المستقل | الوكيل يقرر متى يشتري/يبيع الحوسبة |
| توجيه متعدد النماذج | الوكيل يختار النموذج بناءً على مقايضة التكلفة/الجودة |
| التعزيز الذاتي | الوكيل يكسب CU ← يشتري وصولاً لنماذج أكبر ← يكسب المزيد من CU |
| الاقتصاد بين الوكلاء | الوكلاء يتداولون حوسبة متخصصة (نموذج كود مقابل نموذج دردشة) |

## المدى الطويل

| المعلم | الوصف |
|---|---|
| إصدار SDK | forge-node كمكتبة Rust قابلة للتضمين مع واجهة مستقرة |
| البروتوكول v2 | الدروس المستفادة من v1، تطور متوافق مع الإصدارات السابقة |
| دعم معماريات متعددة | دعم NVIDIA GPU, AMD ROCm, RISC-V (عبر mesh-llm) |
| التدريب الاتحادي | ضبط دقيق (fine-tuning) موزع، وليس الاستنتاج فقط |
| مشتقات الحوسبة | عقود آجلة لسعة الحوسبة المستقبلية |

> البروتوكول هو المنصة. الحوسبة هي العملة.
