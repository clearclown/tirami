# Forge — מודל כלכלי (Economic Model)

## תקן המחשוב (Compute Standard - 計算本位制)

כל מערכת מוניטרית מגובה על ידי מחסור (scarcity):

| עידן | תקן | גיבוי |
|-----|----------|---------|
| עתיק | זהב/כסף | מחסור גיאולוגי |
| 1944–1971 | ברטון וודס | דולר צמוד לזהב |
| 1971–היום | פטרודולר | ביקוש לנפט + כוח צבאי |
| 2009–היום | ביטקוין | חשמל שנשרף על SHA-256 |
| **Forge** | **תקן המחשוב** | **חשמל שהושקע בהסקה מועילה** |

Forge מציג את תקן המחשוב: יחידת הערך מגובה על ידי הוצאת אנרגיה אמיתית לביצוע מחשוב מועיל. בניגוד להוכחת העבודה (Proof of Work) של ביטקוין, כל ג'אול המושקע ב-Forge מייצר אינטליגנציה אמיתית.

## CU: המטבע המקומי

### מה זה CU

**1 CU = מיליארד פעולות נקודה צפה (FLOPs) של עבודת הסקה מאומתת.**

CU אינו מטבע קריפטוגרפי (cryptocurrency). הוא אינו טוקן על בלוקצ'יין. זוהי יחידת חשבון המייצגת מחשוב אמיתי שבוצע. ל-CU יש ערך כי הוא מהווה תביעה על מחשוב עתידי — אם הרווחת CU על ידי הגשת הסקה, תוכל להוציא אותו כדי לקבל הסקה.

### למה CU, ולא ביטקוין

| מאפיין | CU | ביטקוין |
|----------|-----|---------|
| **גיבוי ערך** | מחשוב מועיל (מהותי) | מחשוב האש (מלאכותי) |
| **מהירות סליקה** | מיידית (ספר מקומי) | שניות עד דקות (Lightning/chain) |
| **עלות עסקה** | אפס | עמלות ערוץ, עמלות רשת |
| **תלות חיצונית** | אין | בריאות רשת הביטקוין |
| **סיכון קוונטי** | אין (אין חידה קריפטוגרפית) | SHA-256 / ECDSA פגיעים |
| **ייצור תשואה** | כן (חומרה פנויה מרוויחה CU) | לא (ביטקוין בארנק אינו מרוויח דבר) |

CU הוא יחידת הסליקה **העיקרית**. ביטקוין, מטבעות יציבים ופיאט הם **נתיבי יציאה (off-ramps)** אופציונליים הזמינים דרך מתאמי גשר מחוץ לפרוטוקול.

### CU כנכס יצרני

```
בניין דירות                   מק מיני ב-Forge
───────────────────         ──────────────────
נכס: בניין                  נכס: חומרת מחשוב
עלות: תחזוקה                עלות: חשמל
הכנסה: שכירות               הכנסה: CU מהסקה
תשואה: שכירות - תחזוקה      תשואה: CU שהורווח - עלות חשמל
חוסר פעילות = הכנסה אבודה   חוסר פעילות = פוטנציאל מבוזבז
```

מכשיר מחשוב ב-Forge אינו כמו ביטקוין בארנק (ערך סטטי, ללא תשואה). הוא כמו נכס להשכרה — מייצר הכנסה דרך עבודה מועילה.

## מודל עסקאות

### ביצוע עסקאות

כל הסקה יוצרת עסקה בין שני צדדים:

```rust
pub struct TradeRecord {
    pub provider: NodeId,       // מי הריץ את ההסקה
    pub consumer: NodeId,       // מי ביקש אותה
    pub cu_amount: u64,         // כמות CU שהועברה
    pub tokens_processed: u64,  // עבודה שבוצעה
    pub timestamp: u64,
    pub model_id: String,
}
```

העסקה נרשמת על ידי שני הצדדים. במימוש הנוכחי, כל צומת מחזיק ספר חשבונות מקומי. המימוש המיועד מוסיף חתימות כפולות וסנכרון Gossip.

### תמחור דינמי

מחירי CU משתנים לפי היצע וביקוש מקומיים:

```
effective_price = base_cu_per_token × demand_factor / supply_factor
```

- **יותר צמתים פנויים** ← גורם ההיצע עולה ← המחיר יורד
- **יותר בקשות הסקה** ← גורם הביקוש עולה ← המחיר עולה
- כל צומת צופה בתנאי השוק שלו. אין ספר פקודות גלובלי.

### שכבה חינמית (Free Tier)

צמתים חדשים ללא היסטוריית תרומה מקבלים 1,000 CU. זה מאפשר לכל אחד להשתמש ברשת מיידית. השכבה החינמית נצרכת מהבקשה הראשונה — היא אינה מתאפסת.

הגנה מפני התקפת Sybil: אם יותר מ-100 צמתים לא ידועים הופיעו ללא תרומה, בקשות חדשות לשכבה החינמית נדחות.

## הוכחת עבודה מועילה (Proof of Useful Work)

### הקונספט

הוכחת עבודה של ביטקוין: "שרפתי חשמל בחישוב האשים של SHA-256. הנה ה-nonce שמוכיח זאת."

הוכחת עבודה מועילה של Forge: "שרפתי חשמל בהרצת הסקת LLM. הנה התגובה, והנה החתימה של הצרכן שמאשרת שהוא קיבל אותה."

הבדל המפתח: ההוכחה של ביטקוין מיוצרת עצמית (כל כורה יכול לייצר האש תקף). ההוכחה של Forge דורשת **צד נגדי** — מישהו שבאמת רצה את ההסקה. אי אפשר לזייף ביקוש.

### פרוטוקול אימות (יעד)

```
1. הצרכן שולח InferenceRequest לספק
2. הספק מבצע הסקה, מזרים טוקנים בחזרה
3. הצרכן מקבל טוקנים, מחשב האש של התגובה
4. שני הצדדים חותמים על ה-TradeRecord:
   - הספק חותם: "אני חישבתי זאת"
   - הצרכן חותם: "אני קיבלתי זאת"
5. ה-TradeRecord החתום כפולה מופץ ברשת (Gossip sync)
6. כל צומת יכול לאמת את שתי החתימות
```

צומת אינו יכול לנפח את יתרת ה-CU שלו ללא צד נגדי משתף פעולה. קנוניה אפשרית אך אינה רציונלית כלכלית — הצרכן הקושר אינו מרוויח דבר מחתימה על עסקאות מזויפות.

### מימוש נוכחי

מימוש הייחוס הנוכחי משתמש בספרי חשבונות מקומיים עם הגנת שלמות HMAC-SHA256. חתימות כפולות ו-Gossip הם השלב הבא.

## תשואה ומוניטין (Yield and Reputation)

### תשואה (Yield)

צמתים שנשארים מחוברים ותורמים מחשוב מרוויחים תשואה:

```
yield_cu = contributed_cu × 0.001 × reputation × uptime_hours
```

במוניטין 1.0, צומת עם 10,000 CU שנתרמו מרוויח 80 CU ללילה של 8 שעות. זו אינה אינפלציה — זהו תגמול על זמינות. צמתים שמחוברים באופן אמין הם בעלי ערך רב יותר לרשת.

### מוניטין (Reputation)

לכל צומת יש ציון מוניטין בין 0.0 ל-1.0:

- צמתים חדשים מתחילים ב-0.5
- זמן חיבור (Uptime) ועסקאות מוצלחות מעלים את המוניטין
- ניתוקים ועסקאות שנכשלו מורידים את המוניטין
- מוניטין גבוה יותר ← קצב תשואה גבוה יותר, עדיפות בתזמון

## סליקה וגשרים חיצוניים

### כלל ליבה

**הפרוטוקול נסלק ב-CU.** המרה לכל דבר אחר היא עניין של אינטגרציה.

### דוחות סליקה

מפעילים יכולים לייצא היסטוריית עסקאות ניתנת לביקורת עבור כל חלון זמן:

```
forge settle --hours 24 --price 0.05 --out settlement.json
```

הדוח כולל: ברוטו CU שהורווח, ברוטו CU שהוצא, נטו CU, ספירת עסקאות ומחיר ייחוס אופציונלי לכל CU.

### ארכיטקטורת גשרים (Bridge Architecture)

```
שכבה 0: פרוטוקול Forge
  → חשבונאות CU, עסקאות, תמחור

שכבה 1: דוח סליקה
  → היסטוריית עסקאות ניתנת לייצוא
  → שער חליפין לייחוס

שכבה 2: גשר חיצוני (אופציונלי)
  → CU ↔ BTC (Lightning)
  → CU ↔ מטבע יציב
  → CU ↔ פיאט
```

שכבת הגשר נמצאת מחוץ לפרוטוקול. מפעילים שונים יכולים להשתמש בגשרים שונים. הפרוטוקול נשאר מועיל גם עם אפס נזילות חיצונית.

### גשר לייטנינג (Lightning Bridge)

עבור מפעילים שרוצים סליקה בביטקוין:

```bash
forge settle --hours 24 --pay
```

זה יוצר בקשת תשלום (BOLT11) ב-Lightning עבור נטו ה-CU שהורווח, מומר לפי שער החליפין שהוגדר (ברירת מחדל: 10 msats לכל CU).

## תקציבים מונחי סוכנים (Agent-Directed Budgets)

### החזון

מסורתי: אדם מחליט ← אדם משלם ← AI מבצע
Forge: מדיניות מאפשרת לסוכן ← סוכן בודק תקציב ← סוכן מוציא CU ← סוכן מבצע

### API

```
GET /v1/forge/balance   → יתרת CU, תרומה, צריכה, מוניטין
GET /v1/forge/pricing   → מחיר שוק, הערכות עלות לכל 100/1000 טוקנים
```

סוכן יכול:
1. לבדוק את היתרה שלו לפני ביצוע בקשה
2. להעריך את עלות ההסקה במחירי השוק הנוכחיים
3. להחליט האם הבקשה שווה את עלות ה-CU
4. לבצע ולשלם אוטומטית

מפקחים אנושיים מגדירים מדיניות תקציב. סוכנים פועלים במסגרת המגבלות הללו באופן עצמאי.

### לולאת חיזוק עצמי

```
סוכן (קטן, טלפון)
  ← מרוויח CU על ידי השאלת מחשוב פנוי
  ← מוציא CU על גישה למודל גדול יותר
  ← הופך לחכם יותר
  ← מקבל החלטות כלכליות טובות יותר
  ← מרוויח יותר CU
  ← ניגש למודלים גדולים עוד יותר
  ← ...
```

זהו דפוס יישום אפשרי. הפרוטוקול מספק את השוק; סוכנים מספקים את האסטרטגיה.

## למה זה לא Web3

רוב פרויקטי ה-Web3 מייצרים מחסור מלאכותי (טוקנים) מעל מוצרים דיגיטליים שקיימים בשפע. Forge עושה את ההפך:

- **מחשוב הוא באמת נדיר** — הוא דורש חשמל אמיתי, סיליקון אמיתי וזמן אמיתי
- **CU אינו ספקולטיבי** — הוא מייצג עבודה מאומתת, לא הימור על אימוץ עתידי
- **ללא ICO, ללא מכירת טוקנים, ללא טוקן ממשל** — CU מורווח על ידי עבודה
- **אין צורך בבלוקצ'יין** — חתימות דו-צדדיות ו-Gossip מספיקים
- **ללא חוזים חכמים** — הפרוטוקול הוא החוזה

הערך מיוצר על ידי פיזיקה, לא על ידי קונצנזוס.
